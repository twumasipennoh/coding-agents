### Git branching / history / state-recovery
- **Wrong pitch**: SHA identifiers, "common ancestor," "squash-merged under a different SHA," merge-strategy names, AND stash/pop/HEAD jargon ("stash pop conflict," "WIP captured while on main," "HEAD anomaly," "rebase target for the pop," "git stash pop --merge"). Also wrong: numbered lists of git commands as the plan itself (`git stash push -m "..."`, `git checkout main`, `git fetch origin --prune`, `git pull --ff-only`, `git branch -d`, `git stash pop`) — even a "plan" reply violates the rule when the commands ARE the plan instead of scenario-narrated actions.
- **Right approach**: scenario walkthrough for ALL git situations — sync AND state recovery AND cleanup plans. For sync: what does my local repo think? what does GitHub think? where did they split? what's the fix? Use the "two notebooks" mental model. For state recovery (stash/pop/HEAD movement): what's saved where? what's on the desk now? what was where before? what does the user do if they want X back? Use the "sealed envelope" / "save box" mental model — the saved work is set aside, user can throw it away, unpack carefully, or leave it sealed. For cleanup plans (`/merged`, post-merge branch delete): walk the steps as physical desk motions ("put these edits in a sealed envelope, switch your desk to point at main, pull down GitHub's latest, throw away the fix-branch label, unseal the envelope back onto the desk") — no `git` commands in the plan itself. Commands appear only when kwaku asks for the code-level version.
- **Learned**: 2026-06 — /explain after SHA-heavy diverged-history reply. Recurred 2026-06-23: branch cleanup plan used same jargon. Recurred 2026-06-30: /merged wrap-up used stash/pop/HEAD jargon extending into state-recovery territory. Recurred 2026-07-01: /merged PRE-cleanup plan used a numbered list of raw `git stash` / `checkout` / `pull --ff-only` / `branch -d` / `stash pop` commands as the plan itself. Same calibration rule applies to ALL git mechanics, not just sync — including pre-action plans, not just post-action reports.

### Test fidelity / mock-vs-production accuracy
- **Wrong pitch**: describing the override mechanism ("we set AIDE_MOCK_JSONL_BYTES env var on the mock adapter")
- **Right approach**: three-section coverage walkthrough — (1) what's real (production code paths that fire), (2) what's faked (the single mocked input + why), (3) bug classes caught vs missed. Lead with the production code that's actually getting exercised; the mock is the asterisk, not the headline. One-line analogy ("crash-test ram vs real wall") AFTER the concrete sections, not in place of them.

### Test regression / test-infra failure explanations ("why did they break" / "why did the gate get bypassed")
- **Wrong pitch**: leading with the commit hash + the browser API that changed (IndexedDB, `page.route`, `onAuthStateChanged`, `requestAnimationFrame`), the test-runner internals (`useRealTimers`, `getByTestId`, `storageState`, `startHoldSeek`), or the setup-file name (`.playwright-auth.json`). Reader learns the mechanism but not what happened to their tests.
- **Right approach**: (1) pick one representative test and narrate the developer-run journey before-and-after in plain language — for regressions: "test navigates here, guard asks X, mock returned Y, guard let it through; now guard asks storage instead, mock can't reach it, test lands on login." For infra breakdowns: "setup script normally spins up a fake auth service, signs in a fake user, saves the login keychain to disk; this time the fake auth service never started, so the keychain was never written, so all 112 browsers opened cold and got kicked to login." For flaky-teardown gate bypasses: "these 6 tests need real time to elapse, flip a switch mid-test, then finish without turning the loop off; the loop keeps ticking after the test ends, and the runner marks the whole exit code as unclean." (2) per-layer viability + gap sweep (which layer catches what, which lost coverage, what's now dark), (3) name the missing alarm — why didn't we notice / why did the gate flip green when it should have flipped red. Gaps are the load-bearing part; the mechanism is annotation.
- **Learned**: 2026-07 — /explain after "132 Playwright auth failures" reply led with commit hash + IndexedDB terminology. 2026-07-03 — /explain on F46 bypassed-gates /investigate: reply led with `.playwright-auth.json`, `ProtectedRoute`, `useRealTimers`, `startHoldSeek`, `requestAnimationFrame`. Rule now explicitly covers infra breakdowns and gate-bypass explanations, not only regression walkthroughs.

### Design decisions / option presentation
- **Wrong pitch**: bundled numbered options that conflate independent choices ("Option 1: A+B+C, Option 2: A+D+E"). Also wrong: after listing axes + picks, ending with a vague drill-or-move-on prompt ("Want the next part, or drill in?") instead of naming the concrete action confirmation would trigger. Confirming a "pick" alone doesn't tell the user what gets done if they say "go."
- **Right approach**: present each design axis as a separate choice point — let the user mix and match independently. After listing axes + picks, write a short numbered list of the **concrete actions the picks would trigger** ("Confirm and I'll edit X to do Y, change Z to cite W, etc."). Close with "confirm / swap any pick / drill into one" — the user should be able to type "go" and know exactly what gets done. If picks don't translate to concrete edits, sharpen them first. **Same rule for meta-options / orchestration choices**: "all at once," "bundle with X," "defer to later," "run in parallel" imply an action just like picks do — unpack what each one physically changes (which files get edited once vs. twice, what infrastructure gets combined, what the net delta is) before offering it as a choice.
- **Learned**: 2026-05 — feedback on multi-axis design proposals. 2026-06-30 — /research on auto-calibration ended Q1 with 3 axes + 3 picks then asked "Want Q2, or drill into any Q1 axis?" Kwaku /explain'd asking what he was being asked to confirm. The picks implied actions but the actions were never spelled out. 2026-06-30 (same session) — offered "say 'all at once' to bundle with Q1" without naming what bundling meant. Kwaku /explain'd. Meta-options need the same unpacking as picks.

### Problem-with-mitigations / options-list deliverables

Even if the config trace is technically correct, leading with it makes the user read mechanism before they can feel the failure. Also wrong: inventing concrete-feeling specifics (turn numbers, line ranges, quoted text) to give a scenario narrative texture. Internal contradictions surface immediately and destroy the whole explanation.
- **Right approach**: when the deliverable is "here is a broken thing and N ways to fix it," lead with a **user-journey scenario walkthrough of the root cause** — "you're in a long Telegram thread, every turn aide does X, eventually Y overflows, here's what you saw in your email-automation session at turn 165." Every specific number, quote, and config value in that scenario must come from actually-read evidence in this turn (the session JSON, the code, the agent report). If I don't have the specific, use a generic placeholder ("an earlier rotation event", "much earlier in the session"). Then list the options. Config flags and code paths can appear, but as **annotations on the scenario**, not as the lead. The options list itself follows one-beat-per-turn.

Aide's compaction lives in `~/projects/conversational-assistant/src/operations/session-compaction/` and is triggered independently at 75% of effective-window bytes.

### Multi-part deliverable pacing / "continue" semantics
- **Wrong pitch**: two failure modes on the same rule. (a) **Dumping all parts at once** because "it's a single diagnostic deliverable." A "2 findings + 3 fix options + verdict" reply is a 4+ part answer, not one deliverable. Red-flag opener: "Two problems, not one" (also violates the CLAUDE.md antithesis ban). Red-flag section headers inside a diagnostic reply: `## Options`, `## Two problems`, `## Findings`, `## Fixes`, followed by bulleted alternatives. If I catch myself typing one of these mid-reply, stop and restructure — the header IS the tell. (b) **Replying "No response requested." (or silence)** to "continue" / "continue from where you left off" / "keep going" after I offered a part-1-of-N reply with a "continue?" prompt. Equally wrong: dumping the remaining N-1 parts in one turn once continue is requested.
- **Right approach**: before sending any diagnostic or advisory reply, count the distinct parts (findings, options, caveats, follow-ups). **If ≥3**, restructure: (1) lead with the single most load-bearing part (top-priority finding + its fix) in 1-3 sentences, (2) offer expansion in one short trailing phrase ("also a latent Flask bug + 3 fix options if you want them"), (3) stop. Do NOT use antithesis section headers — they announce the violation. When kwaku replies "continue" / "keep going" / "continue from where you left off," that IS the answer to my expansion offer — deliver the next single beat as its own turn. One option / one finding / one phase per turn. End with a "next is X, continue?" only if more parts remain; on the last part, just close. Bypass phrases for one-shot dump: "all at once" / "dump them all" / "skip." Trust kwaku to steer.
- **Learned**: 2026-06 — /research on aide compaction surfaced 6 options. Delivered Option 1 with "continue with options 2-6?"; kwaku replied "Continue from where you left off"; I replied "No response requested.", silently dropping the deliverable. /explain "with one turn at a time" recalibrated. 2026-07-02 — diagnostic reply on /content/:id "why login prompt?" opened with "Two problems, not one" and dumped 2 findings + 3 fix options + verdict in a single ~500-word turn. kwaku asked "Why did i get this large output" — the diagnosis WAS the rule miss. Rule now covers both upstream (dumping N parts at once, before user asks for continue) and downstream (silently dropping on "continue") failures. Applies across /research, /security-check, /design-check, /memory-review, /feature-status, /investigate, ad-hoc root-cause diagnoses, and fix-advocate delivery.
