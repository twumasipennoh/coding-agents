### Investing / Finance
- **Wrong pitch**: contract/legal jargon, abstract instrument mechanics (pro-rata, dilution, MFN, conversion)
- **Right approach**: first principles with concrete numeric scenarios — dollar amounts, percentages, who-wins-who-loses for each clause
- **Learned**: 2026-05 — reviewing a SAFE, /explain on valuation cap terminology

### Git branching / history / state-recovery
- **Wrong pitch**: SHA identifiers, "common ancestor," "squash-merged under a different SHA," merge-strategy names, AND stash/pop/HEAD jargon ("stash pop conflict," "WIP captured while on main," "HEAD anomaly," "rebase target for the pop," "git stash pop --merge")
- **Right approach**: scenario walkthrough for ALL git situations — sync AND state recovery. For sync: what does my local repo think? what does GitHub think? where did they split? what's the fix? Use the "two notebooks" mental model. For state recovery (stash/pop/HEAD movement): what's saved where? what's on the desk now? what was where before? what does the user do if they want X back? Use the "sealed envelope" / "save box" mental model — the saved work is set aside, user can throw it away, unpack carefully, or leave it sealed
- **Learned**: 2026-06 — /explain after SHA-heavy diverged-history reply. Recurred 2026-06-23: branch cleanup plan used same jargon. Recurred 2026-06-30: /merged wrap-up used stash/pop/HEAD jargon extending into state-recovery territory. Same calibration rule applies to ALL git mechanics, not just sync

### openclaw / aide internals
- **Wrong pitch**: code paths, scanner functions, store names, parser internals ("ScheduleScanner.tick," "parseStreamJsonOutput," "LLMUnavailableError"), Map lookups, field-level guard conditions, line numbers. Also: describing feature-completion benefits in build terms ("parser patterns + executeIntent cases, half a day") instead of what the user gains. Also: writing dedup/uniqueness keys as tuples of internal field names like "(projectId, message, cron, nextFireAtUtcMs)"
- **Right approach**: open with the user-visible action or symptom, trace the chain in observable terms, THEN reference code paths. For tier/fallback systems, frame each tier as "what aide attempts on your behalf at this layer." For catalog/registry bugs, explain as "aide's directory had two entries with the same name, it grabbed the wrong one" not "entries.get() resolved the Smithery entry because !existing.transportKind was false." For "what does shipping X get me" questions AND bug-fix explanations, answer with scenarios: "you said X, aide did Y, here's why, here's what changes". For dedup/uniqueness rules, say plainly "same message at the same time = same reminder; same message at a different time = different reminder," and walk through what happens when the user repeats vs. extends a batch
- **Learned**: 2026-05 — multiple /explain invocations across pipeline, reminder, tier-fallback, and diagnostic-capability discussions. 2026-06 — Gmail OAuth mount bug diagnosed with Map keys and cleanup guard conditions instead of user-journey trace. 2026-06 — patch benefit + catalog seed bug both explained in implementation terms instead of user-experience scenarios. 2026-06 — evaluation risk/mitigation pitched as abstract system concern ("auto-scope mapping incomplete") instead of user failure story. 2026-06 — integration-runner crash diagnosis pitched as "discoverFromDir throws IntegrationRegistryError when pickIntegrationExport returns null" instead of "aide's helper crashes on startup because it trips over a file it shouldn't be loading". 2026-06 — http_request ad-hoc token fix described as "schema field + resolveAdHocAuth helper + credential-resolution gate" instead of "you paste a token, aide uses it for the request". 2026-06 — schedule-dedup batch behavior pitched as tuple-keyed dedup with `nextFireAtUtcMs` and `cron` field names instead of "same message at same time vs. same message at different time". 2026-06-29 — scheduled-task retry-cap root cause delivered with `pending-prompts.ts:2545`, `return null`, `Schedule.consecutiveFailures`, `AUTO_PAUSE_AT_CONSECUTIVE_FAILURES = 2`, `PendingPrompt`, `createSchedule` — instead of "you tell aide to remind you tomorrow at 9am; aide writes a record; scanner wakes every minute and fires due records; on failure it pings you and increments a counter; at 2 strikes it should auto-pause but doesn't, which is the bug." Investigator-agent structured spine output is mechanism-shaped by design — fine for the agent's body, but the parent skill's final delivery should still anchor in scenario when the user follows with /explain. 2026-06-29 — fix-options for the same retry-cap bug delivered with LoC counts, file:line refs, and guard-mechanism descriptions ("sanitize at the dispatcher structured-result paths, ~6 LoC" / "truncate at recordFailure's persist boundary, ~10 LoC") instead of "before: 9 pings every minute; after option A: 2 pings, second one says aide gave up, then silence; after A+B: same plus future-proofing." Fix-option presentation has the same before/after-scenario shape as bug diagnosis — list-of-options with implementation-surface descriptions is the wrong unit. Lead with user-visible behavior under each option, then the implementation surface only if asked. 2026-06-30 — sibling-sweep on the create_field N-yeses bug delivered as labeled bullet categories ("morning standup batch — 4 yeses", "pre-trip prep — 3 yeses", "two-person check-ins — 2 yeses"). kwaku invoked /explain "the scenarios for both that could be annoying." Labels name the bucket but not the felt experience. Right shape: a quoted telegram exchange — kwaku's actual typed message → aide's actual reply → the yes-yes-yes-yes click-through. The annoyance shape is the dialogue, not the category name. 2026-06-30 — same session, recommendation summary between options 1 and 2 for that same bug delivered as labeled reasons ("matches existing codebase pattern / smaller surface to ship / LLM-trust risk is bounded"). kwaku invoked /explain "with scenarios." Same shape miss: a category list is not a recommendation, it's a category list. The right shape for "why pick X over Y" is a quoted telegram exchange for the happy path under X, plus a second exchange for the failure path showing what kwaku sees when X's weakness manifests. ### Mobile keyboard / viewport interactions
- **Wrong pitch**: "blur the input," "debounce," "focus state," "scrollIntoView," "visualViewport API," "scroll-padding-bottom" — code-level mechanics
- **Right approach**: describe what's physically happening on screen — "the keyboard slides up and covers your results." Then what changes: "you stop typing, the keyboard drops on its own so you can see what matched." Walk through the motion under the user's thumb, not the DOM event chain
- **Learned**: 2026-06 — /explain on channel-picker search keyboard fix

### Design decisions / option presentation
- **Wrong pitch**: bundled numbered options that conflate independent choices ("Option 1: A+B+C, Option 2: A+D+E"). Also wrong: after listing axes + picks, ending with a vague drill-or-move-on prompt ("Want the next part, or drill in?") instead of naming the concrete action confirmation would trigger. Confirming a "pick" alone doesn't tell the user what gets done if they say "go."
- **Right approach**: present each design axis as a separate choice point — let the user mix and match independently. After listing axes + picks, write a short numbered list of the **concrete actions the picks would trigger** ("Confirm and I'll edit X to do Y, change Z to cite W, etc."). Close with "confirm / swap any pick / drill into one" — the user should be able to type "go" and know exactly what gets done. If picks don't translate to concrete edits, sharpen them first. **Same rule for meta-options / orchestration choices**: "all at once," "bundle with X," "defer to later," "run in parallel" imply an action just like picks do — unpack what each one physically changes (which files get edited once vs. twice, what infrastructure gets combined, what the net delta is) before offering it as a choice.
- **Learned**: 2026-05 — feedback on multi-axis design proposals. 2026-06-30 — /research on auto-calibration ended Q1 with 3 axes + 3 picks then asked "Want Q2, or drill into any Q1 axis?" Kwaku /explain'd asking what he was being asked to confirm. The picks implied actions but the actions were never spelled out. 2026-06-30 (same session) — offered "say 'all at once' to bundle with Q1" without naming what bundling meant. Kwaku /explain'd. Meta-options need the same unpacking as picks.

### Problem-with-mitigations / options-list deliverables
- **Wrong pitch**: opening with the config flags, ring-buffer sizes, file paths, and field names that produce the failure (`compaction.mode: "safeguard"`, `session.maintenance.mode: "warn"`, "20-message ringBuffer", `~/.openclaw/openclaw.json:95`), then moving to the options list. Even if the config trace is technically correct, leading with it makes the user read mechanism before they can feel the failure. Also wrong: inventing concrete-feeling specifics (turn numbers, line ranges, quoted text) to give a scenario narrative texture. Internal contradictions surface immediately and destroy the whole explanation.
- **Right approach**: when the deliverable is "here is a broken thing and N ways to fix it," lead with a **user-journey scenario walkthrough of the root cause** — "you're in a long Telegram thread, every turn aide does X, eventually Y overflows, here's what you saw in your email-automation session at turn 165." Every specific number, quote, and config value in that scenario must come from actually-read evidence in this turn (the session JSON, the code, the agent report). If I don't have the specific, use a generic placeholder ("an earlier rotation event", "much earlier in the session"). Then list the options. Config flags and code paths can appear, but as **annotations on the scenario**, not as the lead. The options list itself follows one-beat-per-turn.
- **Also**: confirm which SYSTEM owns the config or behavior before citing it. aide (`~/projects/conversational-assistant/`, data at `~/.aide/data/`) and openclaw (`~/.openclaw/`) are TWO different systems with TWO separate compaction implementations. `~/.openclaw/openclaw.json` `compaction.mode: "safeguard"` controls openclaw's gateway-level pruning, not aide's per-project rotation. Aide's compaction lives in `~/projects/conversational-assistant/src/operations/session-compaction/` and is triggered independently at 75% of effective-window bytes.
- **Learned**: 2026-06-30 — /research on aide compaction surfaced 6 fix options. Opened with a config-flag-laden "Root cause of the cliff" paragraph citing the wrong system's config, then dove into the options. kwaku invoked /explain "starting with the root cause before we think about mitigation options." Recalibrated to a scenario-style root cause that included invented turn numbers ("drops everything except turns 146-165. Turn 162, where Option A was named, is now gone") — internally contradictory since 162 is inside 146-165. kwaku flagged the math and the aide/openclaw conflation. Two separate failure modes the calibration now covers: scenario-must-be-grounded-in-evidence, and aide ≠ openclaw.

### Multi-part deliverable pacing / "continue" semantics
- **Wrong pitch**: replying "No response requested." (or silence) to "continue" / "continue from where you left off" / "keep going" after I offered a part-1-of-N reply with a "continue?" prompt. Equally wrong: dumping the remaining N-1 parts in one turn.
- **Right approach**: "continue" IS the answer to my own offer — deliver the next single beat as its own turn. One option / one finding / one phase per turn. End with a "next is X, continue?" only if more parts remain; on the last part, just close. Bypass phrases for one-shot dump: "all at once" / "dump them all" / "skip."
- **Learned**: 2026-06 — /research on aide compaction surfaced 6 options. Delivered Option 1 with "continue with options 2-6?"; kwaku replied "Continue from where you left off"; I replied "No response requested.", silently dropping the deliverable. /explain "with one turn at a time" recalibrated. Applies across /research, /security-check, /design-check, /memory-review, /feature-status, and ad-hoc multi-finding replies.
